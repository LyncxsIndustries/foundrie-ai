## 2. THE DEPENDENCY PROBLEM FORMALLY DOCUMENTED

When AI generates code for a project, it installs a handful of top-level packages. Each of those packages has its own dependencies. The result is that a typical AI-generated Node.js application has hundreds of packages in `node_modules` that the developer has never audited, reviewed, or seen.

**The math:**
- Explicitly installed packages: ~7
- Total installed packages: ~847
- Known vulnerabilities at any given time: potentially 10–20+
- Time most developers spend auditing these: 0 minutes

This is not theoretical risk. Supply chain attacks originate here. Data breaches originate here.

**This is a business risk for Lynxcs Industries.** Shipping a client product with unaudited dependencies is shipping an incomplete product. The dependency audit is as important as the feature implementation.

## 3. THREE-STEP SECURITY PROTOCOL (GENERATED SCAFFOLD)

Foundrie generates this protocol as `docs/SECURITY.md` in every ZIP and embeds it in `context/code-standards.md` as a required developer practice.

### Step 1: Run the Audit Today

```bash
# Node.js projects
npm audit

# Python projects
pip-audit
# or
safety check

# Auto-fix what is safe to fix
npm audit fix

# Review what cannot be auto-fixed
npm audit fix --dry-run
```

This single command reveals: total vulnerability count, severity breakdown (critical/high/medium/low), which packages are causing them, and whether a fix is available.

**Generated in `docs/SECURITY.md`:**
```markdown
## Initial Audit

Run `npm audit` immediately after setup. Do not proceed to feature
implementation until you have reviewed the audit output and resolved
all critical and high-severity findings.

Many developers have never run this command once. What you see the
first time will be surprising. This is normal. Run it and fix it.
```

### Step 2: Lock Your Versions

The lock file (`package-lock.json` for Node, `poetry.lock` for Python, `Cargo.lock` for Rust) is an insurance policy.

**Rules generated in `context/code-standards.md`:**
```markdown
## Lock File Rules

- Never delete the lock file
- Never add the lock file to .gitignore
- Always commit the lock file to the repository
- Treat changes to the lock file as requiring code review
- Every CI build uses the lock file (npm ci, not npm install)

Without a lock file, npm install on different machines at different
times resolves to different package versions. This creates bugs that
are essentially impossible to reproduce and security regressions that
cannot be tracked.
```

**Generated `.npmrc` or `pip.conf`** with version pinning preferences:
```ini
# .npmrc — generated by Foundrie
save-exact=true
engine-strict=true
```

**Generated `package.json` scripts:**
```json
{
  "scripts": {
    "install:ci": "npm ci",
    "audit:check": "npm audit --audit-level=moderate",
    "audit:fix": "npm audit fix",
    "outdated:check": "npm outdated"
  }
}
```

### Step 3: Monthly Audit Cadence

Generated in `docs/SECURITY.md`:

```markdown
## Monthly Audit Protocol

First Monday of every month — 10-minute block:

1. npm audit           → check current vulnerability state
2. npm outdated        → see what has updates available
3. Update patch versions of trusted dependencies
4. Flag major version updates for manual review
5. Log results in CHANGE_LOG.md under "Security Audits"

10 minutes per month is the difference between a secure product
and a data breach. Schedule it. Do it. Log it.

Compliance record format:
  Date: [YYYY-MM-DD]
  Auditor: [name]
  Critical CVEs: [count]
  High CVEs: [count]
  Action taken: [description]
  Next scheduled: [YYYY-MM-DD]
```

## 4. DEPENDABOT AND SBOM GENERATION

### Generated `.github/dependabot.yml`

```yaml
# .github/dependabot.yml — generated by Foundrie in every ZIP
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
      time: "09:00"
    open-pull-requests-limit: 5
    reviewers:
      - "team-lead"
    labels:
      - "dependencies"
      - "security"
    commit-message:
      prefix: "chore(deps)"
    ignore:
      - dependency-name: "*"
        update-types: ["version-update:semver-major"]

  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "monthly"
```

Dependabot opens automated PRs for dependency updates. Major version bumps are excluded (those require manual review and ADR documentation). Security patches are merged automatically if CI passes.

### SBOM (Software Bill of Materials)

Generated as a CI step in the release pipeline:

```yaml
# .github/workflows/release.yml — SBOM generation added
- name: Generate SBOM
  uses: anchore/sbom-action@v0
  with:
    format: "spdx-json"
    output-file: "sbom.spdx.json"
    artifact-name: "sbom"
```

The SBOM documents every dependency at every transitive level. It is attached to every GitHub Release artifact. This enables compliance reporting and rapid response when a new CVE is disclosed — the SBOM answers immediately: "does our product use the affected package?"

## 6. CLIENT LIFECYCLE MANAGEMENT — FIVE PHASES

Foundrie and Ruwa operate across the full client relationship lifecycle. The following framework is the operational standard for all Lynxcs Industries client work.

```
Phase 1: DISCOVERY & QUALIFICATION
Phase 2: ONBOARDING
Phase 3: ACTIVE DELIVERY
Phase 4: RETAINMENT
Phase 5: OFFBOARDING
```

Each phase has defined Foundrie generation responsibilities, Ruwa research responsibilities, and completion gates that must be cleared before advancing.

---

## 7. PHASE 1: DISCOVERY & QUALIFICATION

**Goal**: Determine if the client is a fit, understand their needs, set expectations.

### Qualification Gate (Five Required Questions)

Before any proposal is generated, Foundrie conducts a qualification check:

1. What is the specific problem they need solved?
2. What is their timeline and budget?
3. Who is the decision-maker (the person who can say yes)?
4. Do they have the infrastructure to receive and use what we build?
5. Are their expectations aligned with what's achievable in their timeline?

If any question cannot be answered confidently: do not generate a proposal yet. Ruwa conducts additional research on the prospective client to fill the gaps.

### Ruwa's Role in Phase 1

- Research the prospective client's industry, key competitors, current digital presence.
- Identify pain points their business is likely experiencing (based on industry benchmarks and their public positioning).
- Benchmark pricing for their type of project using market data.
- Produce a "Discovery Brief" that informs the Foundrie-generated proposal.

### Foundrie's Role in Phase 1

- Generate a tailored Collaboration Proposal (§12.2) using the Discovery Brief as context input.
- Generate an NDA (§12.5) if the discovery conversation involves sensitive information.

### Discovery Call Agenda (Generated by Ruwa)

```
1. Client context (5 min) — understand their business and goals
2. Pain point articulation (10 min) — what is broken or missing?
3. Scope discussion (10 min) — what does a solution look like?
4. Timeline and budget alignment (5 min)
5. Next steps (5 min) — proposal timeline, NDA if needed
```

## 8. PHASE 2: ONBOARDING

**Goal**: Bring the client from "signed" to "active and aligned" with minimal friction.

### Onboarding Kit (Generated by Foundrie)

Foundrie generates a complete onboarding kit document containing:
- Welcome message with project overview and context
- Shared workspace setup instructions (Google Drive, Notion, or equivalent)
- Communication protocol: who to contact, for what, via which channel, response SLA
- Revision policy and approval workflow (number of included rounds, cost for additional rounds)
- Payment schedule reminder with due dates

### Ruwa's Role in Phase 2

Produces a "Client Intelligence Brief" — a structured research document covering:
- The client's domain, audience, competitive landscape, and recent industry developments
- Relevant platform changes or algorithm updates that affect their project
- Market rate benchmarks for their project type (validates the agreed pricing)

This brief is stored in the client's Ruwa knowledge base (Mem0 — v12.0.0 §4) and is retrieved automatically in future sessions for this client.

### Onboarding Checklist (Eight Items — Hard Gate)

```
[ ] NDA signed and filed
[ ] Creator Agreement / Service Agreement signed and filed
[ ] Upfront payment received and confirmed
[ ] Shared workspace created and access granted
[ ] Communication channel established (email, Slack, WhatsApp)
[ ] Project timeline shared, reviewed, and approved in writing
[ ] Revision limits and scope communicated in writing
[ ] Client intelligence brief completed (Ruwa)
```

**Critical Onboarding Principle**: Set constraints early. Clients who are not informed about revision limits, scope boundaries, and timeline dependencies become the clients who cause delays, disputes, and scope creep. Every constraint is documented in writing before work begins — not after a dispute arises.

## 9. PHASE 3: ACTIVE DELIVERY

**Goal**: Deliver the agreed scope on time, at agreed quality, with clear communication.

### Delivery Principles (Operational Standards)

```
1. Every delivery has a version number: "Draft 1", "Draft 2", etc.
   Never deliver without a version. Never deliver a "final" without
   the client having approved an intermediate draft.

2. Every revised delivery includes a "What Changed" note.
   One paragraph. Not optional.

3. Never ask open-ended questions to a client.
   Wrong: "What do you think?"
   Right: "We can go with Option A (darker background) or Option B
           (lighter background) — which fits your brand better?"

4. All approval confirmations are in writing.
   "Looks good, thanks!" in a WhatsApp message counts.
   Log it. Screenshot it. Keep it.
```

### Ruwa's Role During Active Delivery

- Fact-check any deliverable content that makes empirical claims.
- Research any new requirements that arise mid-project.
- Monitor for external changes (platform updates, algorithm changes, policy shifts) that affect the work in progress.
- Alert the engineer immediately when an external change requires adapting the deliverable.

### Foundrie's Role During Active Delivery

- Maintain version-controlled document outputs (each version stored separately in the client folder per §13 storage protocol).
- Generate revision logs and change summaries for each new version.
- Produce interim progress reports for projects exceeding two weeks in duration.
- Generate a Scope Amendment document for every out-of-scope client request.

### Scope Amendment Document

When a client requests something outside the original scope:

```markdown
# Scope Amendment — [Project Name]

**Date**: [date]
**Requested by**: [client name]
**Original scope reference**: [Agreement date or document number]

## New Request Description
[Precise description of what the client is requesting]

## Additional Cost
[Amount] [Currency] — [payment timing]

## Impact on Timeline
+[N] days from original completion date
New estimated completion: [date]

## Approval Required
This amendment becomes effective upon written approval.
Work on the amended scope does not begin before approval is received.

Approved by: ________________  Date: ________________
```

No work begins on an out-of-scope request without a signed Scope Amendment. This is non-negotiable.

## 10. PHASE 4: RETAINMENT

**Goal**: Convert one-time clients into recurring revenue.

### Value-Add Check-Ins (Ruwa-Powered)

Ruwa generates a monthly "industry intelligence brief" for retained clients:
- Alerts to platform changes, new opportunities, or risks in their niche.
- Competitive intelligence (what competitors are doing that the client should know about).
- Relevant case studies or research relevant to their ongoing projects.

This positions Lynxcs Industries as a strategic partner — not just a vendor — and creates a natural reason to stay in contact without it feeling like a sales call.

### Retainer Package Structure (Generated by Foundrie)

```
RETAINER PROPOSAL COMPONENTS:
  - Monthly deliverable bundle (defined, not open-ended)
  - Priority access to Foundrie + Ruwa capabilities
  - Dedicated monthly support window (defined hours)
  - Ongoing monitoring and alerting (Ruwa's intelligence function)
  - Discounted rate vs. ad-hoc project pricing (% specified)
  - Rollover policy (unused hours: do they roll or expire?)
```

### Three Retainment Triggers

Ruwa flags these conditions as retainment opportunities:

1. **30 days post-project completion** — follow-up check-in with the client.
2. **Major change detected in the client's industry** — Ruwa alerts, engineer reaches out with relevant intelligence.
3. **New Lynxcs capability launch** — when a new Foundrie or Ruwa capability is relevant to the client's previous project, proactive outreach is appropriate.

### Client Health Scoring (Three Dimensions)

```
RESPONSIVENESS:    Are they engaging with check-ins?
                   Metric: average reply time to communications.
                   Green = < 48h. Yellow = 48–96h. Red = > 96h or no reply.

SATISFACTION:      Did they provide positive feedback or referrals?
                   Metric: NPS or direct testimonial provided. Referrals made.
                   Green = testimonial given. Yellow = positive but no testimonial.
                   Red = no feedback or expressed dissatisfaction.

FIT:               Are their growing needs still aligned with our capabilities?
                   Metric: does their next project fall within Lynxcs' service scope?
                   Green = strong alignment. Yellow = partial. Red = out of scope.

Overall health = lowest score across all three dimensions.
Red on any dimension = immediate human review required.
```

```
[ ] All deliverables transferred and client confirmed receipt in writing
[ ] Final payment received and confirmed

[ ] License terms documented, confirmed, and filed
```

### Handling Early Terminations

If a client terminates before project completion, the Creator Agreement (§12.4) governs:
- Work completed to date is invoiced at the agreed per-milestone rate.
- Upfront deposits are non-refundable unless the Creator Agreement explicitly states otherwise.
- All work product created up to termination is delivered.
- A final accounting document is generated by Foundrie showing: work completed, payment received, final amount owed (or refunded).

Early termination is handled in writing only. Verbal agreements about termination terms are not enforceable.

## 12. FOUNDRIE-GENERATED CLIENT DOCUMENTS

All Lynxcs Industries client documents are generated by Foundrie. Every document must meet four quality criteria before delivery:

1. **Completeness** — No empty placeholders. Every field populated or explicitly flagged for human review.
2. **Legal coherence** — Clear jurisdiction language, enforceable payment schedules, unambiguous terms.
3. **Brand alignment** — Reflects Lynxcs Industries' professional positioning.
4. **Actionability** — The recipient can act on the document immediately without asking for clarification.

### 12.1 Rate Card

**Purpose**: Communicate Lynxcs Industries' pricing clearly and professionally to prospective clients.
**When generated**: Before or during Phase 1 (Discovery). Updated annually.

**Generation Rules:**
- Always include a "Valid for [Year]" header. Rates change; documents should not persist indefinitely.
- Express licensing as a percentage add-on, never a flat fee (allows flexibility per deal).
- Never publish internal cost margins (reveals profitability data).
- The rate card is a marketing document as much as a pricing document — format and tone must reflect this.

**Required Sections:**
```
RATE CARD — Valid for [Year]
Prepared by Lynxcs Industries

ABOUT
Entity: Lynxcs Industries / Lynxcs Studios
Specialization: [description]
Primary Channels: [platforms]
Total Reach: [aggregate audience]

INDIVIDUAL DELIVERABLES
Format / Platform | Scope | Rate (KES)
──────────────────────────────────────
[Row per deliverable type]

CAMPAIGN PACKAGES
Package Name | Deliverables | Rate (KES)
──────────────────────────────────────
Starter | [contents] | [amount]
Scale   | [contents] | [amount]

LICENSING & ADD-ONS
Right | Rate
──────────────────────────────────────
Paid Amplification (30 days) | +X% of base fee
Organic Repost (3 months)    | +X% of base fee
Exclusivity (per category)   | +X% of base fee
```

### 12.2 Collaboration Proposal

**Purpose**: A tailored document presenting a specific campaign or project offering to a prospective client.
**When generated**: End of Phase 1, before contract signing.

**Generation Rules:**
- The proposal must reflect the client's language and stated goals — not generic agency copy.
- Every deliverable must include execution detail, not just a label.
- The timeline must be realistic. Under-promise, over-deliver.
- Payment terms must be explicit (50/50 is standard; 100% upfront for small or new clients).

**Foundrie + Ruwa Collaboration Model:**
Ruwa researches the client's industry context and produces a Discovery Brief before Foundrie generates the proposal. The Campaign Objective section references the client's competitive landscape, audience behavior, or industry trends — it is not generic filler.

**Required Sections:**
```
1. Campaign Objective (customized per client — informed by Ruwa's research)
2. Scope of Work (specific deliverables with execution detail)
3. Production Timeline (milestone-based, not vague date ranges)
4. Commercial Terms and Licensing
5. Total Investment
6. Next Steps (what happens after they sign)
```

### 12.3 Invoice

**Purpose**: Legally compliant billing document for services rendered.
**When generated**: Upon milestone completion or project completion.

**Invoice Numbering Format**: `LI-[YEAR]-[SEQUENCE]`
Example: `LI-2026-047`

**Generation Rules:**
- Sequence number is maintained in an invoice log per client.
- Payment due date is always 14 days from invoice date (stated explicitly).
- TDS note is always included for Indian clients — the brand deducts TDS and must provide a certificate.
- All amounts maintain two decimal places throughout. No inconsistent rounding.
- Tax line clearly states the rate and whether it is inclusive or exclusive.

**Per-Client Invoice Log (Maintained by Foundrie):**
```
Invoice Number | Date Issued | Amount | Status | Payment Reference
LI-2026-047    | 2026-05-21  | KES X  | ISSUED | —
LI-2026-048    | 2026-06-04  | KES Y  | PAID   | MPESA-REF-123
```

### 12.4 Creator Agreement

**Purpose**: The binding legal document governing what is produced, who pays, and what rights transfer.
**When generated**: After proposal acceptance, before work begins.

**Required Sections:**
```
1. Scope of Work — exact deliverables, formats, platforms
2. Commercial Terms — total compensation, payment schedule, TDS responsibilities
3. Content Ownership & Licensing — creator retains ownership; client gets defined license
4. Revisions & Exclusivity — included revision rounds, costs for additional rounds, exclusivity terms
5. Dispute Resolution — jurisdiction and governing law
```

**Three Non-Negotiable Principles:**

**On Ownership**: Lynxcs (Creator) retains ownership of all produced content. The client receives a license — not ownership. Transferring ownership requires a separate, higher-fee agreement. This is not negotiable.

**On Revisions:**
```
Revision Billing Standard:
  Rounds 1–2 (Included):  No charge
  Round 3:                Invoice at 15% of original project fee
  Round 4+:               Invoice at 20% of original project fee per round
  Full Reshoot:           Invoice at 60–80% of original project fee

"Minor edits" definition (generated into the agreement):
  Minor = no new concepts, no reshoots, text and color adjustments only.
  New concepts or reshoots are not minor — they are billed as full rounds.
```

**On Exclusivity**: Category exclusivity (preventing Lynxcs from working with a client's competitors for a defined period) commands a premium and must be time-bounded. Unlimited or indefinite exclusivity is never offered.

**On Jurisdiction**: Default jurisdiction is Nairobi, Kenya. Adapt as needed for international clients; document the adaptation in the agreement header.

### 12.5 Non-Disclosure Agreement (NDA)

**Purpose**: Protect confidential information shared during the collaboration process.
**When generated**: Before any sensitive information is shared — ideally before the first detailed discovery call.

**Six Categories of Covered Information:**
1. Unreleased product details and campaign strategies
2. Pricing structures and payment terms
3. Scripts, creative briefs, and content concepts
4. Audience analytics and performance data
5. Internal business operations and processes
6. Technical architecture and implementation details

**Key NDA Terms:**
- Duration: 2 years from signing (standard)
- Obligations: Confidentiality, no third-party disclosure, use only for the stated collaboration
- Exclusions: Information already public, independently known, or legally obtained from a third party
- Jurisdiction: Courts of Nairobi, Kenya (or client's jurisdiction where required by law)

**Four-Step NDA Workflow:**
```
1. Foundrie generates NDA with correct party details
2. Both parties sign (DocuSign, Adobe Sign, or PDF with wet signature scan)
3. Copy retained by Lynxcs Industries in client file
4. NDA reference number noted in the client's master agreement folder
```
