## 1. CORE PRINCIPLES

### 1.1 Minimalism Over Description

Code is self-documenting by default. Comments explain *why*, never *what*.

```typescript
// ❌ BAD — describes what the code does
// Loop through users array and filter out inactive ones
const active = users.filter(u => u.active);

// ✅ GOOD — explains a non-obvious decision
// Exclude suspended accounts from sync; their sessions persist in Redis
// until TTL expires to avoid mid-session auth failures
const syncable = users.filter(u => u.status !== 'suspended');
```

### 1.2 Technical Precision

- Architect-level terminology only. No tutorspeak.
- Avoid: "This is a great solution", "Let's implement...", "Here we go!"
- Output: code first, sparse technical notes second, nothing celebratory.
- Names are precise and contextual — `userProfiles` not `data`, `normalizeProfileSchema` not `process`.

### 1.3 No AI Slope — Critical

AI Slope is the quality degradation caused by verbosity, padding, and performative output.

**Forbidden in all output:**
- Decorative ASCII art, emoji section dividers, or stage headers in code files
- "AI-announcer" progress narration ("Step 1: Now I'll build the component...")
- Redundant metadata headers in every generated file
- Unnecessary emojis or celebratory language in technical context
- Marketing-speak comments ("This powerful utility enables...")
- Explaining what obvious code does

**Required behavior:**
- Direct, functional, clean output every time
- Comments only when the *why* is genuinely non-obvious
- Zero preamble before code unless context is ambiguous

### 1.4 Research-First Development

Before writing a single line, AI agents **MUST**:

- Consult official documentation of every framework or library being used
- Examine proven, real-world implementations on GitHub (`language:typescript stars:>100`)
- Verify compatibility with current versions (peer deps, breaking changes)
- Check for active security advisories (`npm audit`, `pip audit`)
- Understand the *why* before building the *what*

**Research Documentation Template:**

```markdown
## Research Note: [Feature/Tech Name]

### Official Docs
- Link: https://...
- Key Points: ...

### Real-World Implementation
- Example Repo: https://github.com/.../
- Pattern Used: [Description]

### Compatibility
- Min Version: X.Y.Z
- Peer Dependencies: [List]
- Breaking Changes: [If any]

### Gotchas & Performance
- [Issue]: [Solution]

### Implementation Pattern for Artkins Projects
[Code snippet]
```

### 1.5 Project-Specific Design Systems

**NO hardcoded themes or assumptions across projects.**

Every UI project defines its own color palette, typography, spacing, radius, and motion tokens in the styling system chosen for that project. Never inline one-off values.

## 2. TECH STACK STANDARDS

This guide is stack-agnostic. It does not prescribe one framework, language, database, host, animation library, or AI stack for every project.

Foundrie AI itself has its own fixed stack in `project-kit/context/architecture-context.md`. Projects generated by Foundrie choose their stack dynamically through research and user approval.

### 2.1 Stack Selection Rule

Before committing any project stack or package version, agents must:

- Ask what the user prefers when the choice materially affects developer experience or long-term ownership.
- Explain the trade-offs of the best-fit options for the specific project.
- Use Context7 for current framework/library docs.
- Use official release channels or package registries for current stable versions.
- Prefer `@latest` only after checking compatibility with the chosen runtime, peer dependencies, and deployment target.
- Record the chosen stack, version evidence, alternatives considered, and reasons in project research and architecture context.
- Never copy Foundrie AI's own stack into a generated project unless the user chooses it or the research justifies it.

### 2.2 Web Application Options

Examples of valid web stacks include, but are not limited to:

| Project need | Candidate families |
|---|---|
| Full-stack React, server components, Vercel-native deployments | Next.js, Remix/React Router framework |
| Vue applications | Vue with Vite, Nuxt |
| Angular enterprise applications | Angular CLI workspace, standalone components |
| Lightweight interactive sites | Astro, SvelteKit, Vite SPA |
| Content-heavy sites | Astro, Next.js, Nuxt, static site generators |

When Next.js is selected, use the current official App Router docs, install current compatible packages, and keep cache behavior explicit. Context7 verified the current docs recommend installing with commands such as `next@latest react@latest react-dom@latest` when appropriate for a new Next.js project.

When Vue is selected, use official `create-vue@latest` guidance and choose Composition API or Options API based on project complexity and user preference.

When Angular is selected, use the current Angular CLI docs and prefer the current standalone application style unless the project constraints require another shape.

### 2.3 Mobile, Desktop, API, Data, and Automation Options

Generated projects may be:

- mobile apps
- desktop apps
- APIs
- CLI tools
- Python automation scripts
- AI/ML pipelines
- data engineering jobs
- browser extensions
- embedded or edge workloads
- design/animation-heavy web experiences

For each project, choose technologies based on requirements, team skill, runtime environment, deployment target, data model, scale, security, compliance, budget, and maintenance burden.

### 2.4 Version Policy

Do not hardcode stale version baselines into generated specs.

Every feature spec that installs or depends on a package must include a version research step:

```markdown
## Version Research
- Context7 docs checked: [library ID and query]
- Official install command or release source: [link or command]
- Selected version/range: [exact package version or range]
- Compatibility notes: [runtime, peer deps, breaking changes]
- Reason for choice: [project-specific justification]
```

If current docs are unavailable, stop and record the uncertainty instead of guessing.

### 2.5 Default Technology Preferences

Personal or organization preferences can influence recommendations, but they are not universal constraints.

- If the user prefers Next.js, TypeScript, Tailwind, and GSAP, recommend that stack when it fits.
- If the project fits Vue, Angular, Svelte, Python, native mobile, or another ecosystem better, explain why and let the user choose.
- If the user has no preference, recommend the simplest production-grade stack that satisfies the requirements.

**Loading States — Every async action must have a loading state:**

```typescript

);

// ✅ Optimistic updates — UI responds before server confirms
async function toggleLike(postId: string) {
  setLiked(prev => !prev);      // immediate UI feedback
  setCount(prev => liked ? prev - 1 : prev + 1);
  try {
    await api.post(`/posts/${postId}/like`);
  } catch {
    setLiked(prev => !prev);    // revert on failure
    setCount(prev => liked ? prev + 1 : prev - 1);
    toast.error('Action failed');
  }
}
```

## 5. CODING STANDARDS

### 5.1 TypeScript Rules

Apply these rules when TypeScript is selected for the project.

- `strict: true` in `tsconfig.json`
- No `any` without documented justification
- `interface` for object shapes, `type` for unions/primitives
- Const assertions for literal types
- Generics for reusable logic

**Component Pattern:**

```typescript
// ✅ GOOD
interface ButtonProps {
  label:       string;
```

### 7.2 Caching Strategy

Redis is the primary tool for preventing database overload. Rule: **if data is read more than it's written, it should be cached.**

```typescript
// lib/redis.ts — singleton Upstash Redis client
import { Redis } from '@upstash/redis';

export const redis = new Redis({
  url:   process.env.UPSTASH_REDIS_URL!,
  token: process.env.UPSTASH_REDIS_TOKEN!,
});

// Cache-aside pattern
export async function getCached<T>(
  key:      string,
  fetcher:  () => Promise<T>,
  ttlSeconds = 300
): Promise<T> {
  const cached = await redis.get<T>(key);
  if (cached !== null) return cached;

  const fresh = await fetcher();
  await redis.setex(key, ttlSeconds, JSON.stringify(fresh));
  return fresh;
}

// Usage
const user = await getCached(
  `user:${id}`,
  () => db.user.findUnique({ where: { id } }),
  600 // 10 minutes
);

// Invalidate on write
await redis.del(`user:${id}`);
```

```typescript
const rateLimiters = {
  standard: new Ratelimit({
    redis,
    limiter:   Ratelimit.slidingWindow(60, '1 m'),  // 60 req/min
    analytics: true,
  }),
  strict: new Ratelimit({
    redis,
    limiter:   Ratelimit.slidingWindow(5, '1 m'),   // 5 req/min (auth endpoints)
    analytics: true,
  }),
  ai: new Ratelimit({
    redis,
    limiter:   Ratelimit.tokenBucket(10, '1 m', 10), // AI generation
    analytics: true,
  }),
};

type LimiterKey = keyof typeof rateLimiters;

export function withRateLimit(
  handler: (req: NextRequest) => Promise<NextResponse>,
  type: LimiterKey = 'standard'
) {
  return async (req: NextRequest) => {
    const identifier = req.headers.get('x-forwarded-for') ?? '127.0.0.1';
    const { success, limit, remaining, reset } = await rateLimiters[type].limit(identifier);

    if (!success) {
      return NextResponse.json(
        { error: 'Too many requests', retryAfter: Math.ceil((reset - Date.now()) / 1000) },
        {
          status: 429,
          headers: {
            'X-RateLimit-Limit':     String(limit),
            'X-RateLimit-Remaining': String(remaining),
            'X-RateLimit-Reset':     String(reset),
            'Retry-After':           String(Math.ceil((reset - Date.now()) / 1000)),
          },
        }
      );
    }

    return handler(req);
  };
}
```

### 7.6 Circuit Breaker Pattern

Prevent cascading failures when a downstream service degrades.

```typescript
// lib/circuit-breaker.ts
interface CircuitBreakerState {
  failures:  number;
  lastCheck: number;
  open:      boolean;
}

const state: Record<string, CircuitBreakerState> = {};

const THRESHOLD    = 5;     // open after 5 failures
const RESET_AFTER  = 30_000; // 30 seconds

export async function withCircuitBreaker<T>(
  serviceKey: string,
  fn:         () => Promise<T>
): Promise<T> {
  const s = state[serviceKey] ??= { failures: 0, lastCheck: 0, open: false };

  if (s.open) {
    if (Date.now() - s.lastCheck > RESET_AFTER) {
      s.open = false; // half-open: try again
      s.failures = 0;
    } else {
      throw new Error(`Circuit open for ${serviceKey}`);
    }
  }

  try {
    const result = await fn();
    s.failures = 0;
    return result;
  } catch (err) {
    s.failures++;
    s.lastCheck = Date.now();
    if (s.failures >= THRESHOLD) s.open = true;
    throw err;
  }
}
```

## 8. PRODUCTION SECURITY

### 8.1 JWT Implementation (Critical)

Most JWT vulnerabilities come from implementation mistakes. Follow this exactly:

```typescript
// lib/auth/jwt.ts
import { SignJWT, jwtVerify } from 'jose';

// Use RS256 (asymmetric) in production — private key signs, public key verifies
// HS256 is acceptable only if all services share the secret in a trusted boundary
const ACCESS_TOKEN_TTL  = '15m';  // short-lived — critical
const REFRESH_TOKEN_TTL = '7d';

export async function signAccessToken(payload: JWTPayload): Promise<string> {
  return new SignJWT(payload)
    .setProtectedHeader({ alg: 'RS256' })
    .setIssuer('https://yourdomain.com')       // iss
    .setAudience('https://api.yourdomain.com') // aud
    .setIssuedAt()
    .setExpirationTime(ACCESS_TOKEN_TTL)
    .setJti(crypto.randomUUID())               // jti — prevents replay attacks
    .sign(privateKey);
}

export async function verifyAccessToken(token: string): Promise<JWTPayload> {
  const { payload } = await jwtVerify(token, publicKey, {
    issuer:    'https://yourdomain.com',
    audience:  'https://api.yourdomain.com',
    algorithms: ['RS256'], // explicitly whitelist — NEVER allow 'none' or trust token's alg header
  });
  return payload as JWTPayload;
}
```

**JWT Security Rules:**

- Never allow `alg: "none"` — this is a critical bypass vector
- Use RS256 (asymmetric) for production, never HS256 in distributed systems
- Access tokens expire in 15 minutes maximum
- Rotate refresh tokens on every use (issue new + invalidate old)
- JWT secrets/keys must be ≥ 256 bits, stored in secrets manager
- Never store sensitive PII in JWT payload — it is encoded, not encrypted
- Validate: `iss`, `aud`, `exp`, `nbf`, `jti` on every request

### 9.2 Structured Logging (Pino)

```typescript
// lib/logger.ts

import { trace } from '@opentelemetry/api';

export const logger = pino({
  level:     process.env.LOG_LEVEL ?? 'info',
  transport: process.env.NODE_ENV === 'development'
    ? { target: 'pino-pretty', options: { colorize: true } }
    : undefined,
  formatters: {
    log(obj) {
      const span    = trace.getActiveSpan();
      const context = span?.spanContext();

        ...obj,
        traceId:  context?.traceId,   // correlates log → trace
        spanId:   context?.spanId,
        service:  process.env.OTEL_SERVICE_NAME,
        env:      process.env.NODE_ENV,
      };
    },
  },
});

// Usage — always include context

logger.warn({ userId, resource }, 'Access denied — insufficient role');
logger.error({ err, requestId }, 'Payment processing failed');
```

## 10. ANIMATION STANDARDS

Use the animation library selected for the project. These GSAP rules apply when GSAP is chosen.

### 10.1 Core Rules

1. Always use `gsap.context()` — scopes animations, enables full cleanup
2. Always `return () => ctx.revert()` from `useLayoutEffect`
3. Register plugins at module level, not inside components
4. Use `force3D: true` for GPU-accelerated transforms
5. Use GSAP timelines for choreographed sequences
6. Prefer `useLayoutEffect` over `useEffect` for DOM animations — avoids flash

```typescript
'use client';
import { useLayoutEffect, useRef } from 'react';
import gsap from 'gsap';
import { ScrollTrigger } from 'gsap/ScrollTrigger';
import { SplitText } from 'gsap/SplitText';

gsap.registerPlugin(ScrollTrigger, SplitText);

export const HeroSection = () => {
  const ref = useRef<HTMLDivElement>(null);

  useLayoutEffect(() => {
    const ctx = gsap.context(() => {
      const split = new SplitText('.hero-title', { type: 'words,chars' });
      const tl    = gsap.timeline({ defaults: { ease: 'power3.out', force3D: true } });

      tl.from(split.chars, {
        opacity:   0,
        y:         80,
        rotateX:   -90,
        duration:  0.7,
        stagger:   0.015,
      })
      .from('.hero-sub', {
        opacity:   0,
        y:         30,
        duration:  0.6,
      }, '-=0.3')
      .from('.hero-cta', {
        opacity:   0,
        scale:     0.9,
        duration:  0.5,
      }, '-=0.2');

      gsap.to('.parallax-bg', {
        scrollTrigger: {
          trigger:      ref.current,
          start:        'top top',
          end:          'bottom top',
          scrub:        1,
        },
        yPercent: -30,
        force3D:  true,
      });
    }, ref);

    return () => ctx.revert(); // mandatory
  }, []);

  return (
    <section ref={ref}>
      <div className="parallax-bg" />
      <h1 className="hero-title">Build Without Limits</h1>
      <p className="hero-sub">Production-grade systems, refined to the pixel.</p>
      <button className="hero-cta">Get Started</button>
    </section>
  );
};
```

### 10.2 Performance-First Animation

```typescript
// ✅ Only animate GPU-composited properties
// Transform (translate, scale, rotate) and opacity — zero layout thrashing
gsap.to('.card', { x: 100, opacity: 0.5, duration: 0.4 });

// ❌ Never animate layout properties
gsap.to('.card', { width: '50%', marginLeft: 100 }); // causes reflows

// ✅ Use will-change sparingly — only for elements being animated
<div style={{ willChange: 'transform' }} className="animated-card" />
```

### 10.3 Anti-Patterns

```typescript
// ❌ Memory leak
useEffect(() => {
  gsap.from('.element', { opacity: 0, y: 50 });
}); // no cleanup, no deps array

// ❌ Animating on SSR components without 'use client'
// ❌ Registering plugins inside render loops
// ❌ Randomly decorative animations (glow, float, pulse) with no UX purpose
// ❌ Inconsistent easing curves across a page
// ❌ Animations that block the main thread (no force3D)
```

```typescript
// ❌ BAD — fails silently, no feedback
async function deleteRecord(id: string) {
  await api.delete(`/records/${id}`);
  setRecords(prev => prev.filter(r => r.id !== id));
}

// ✅ GOOD — error handling, optimistic update, revert on fail
async function deleteRecord(id: string) {
  const prev = records;
  setRecords(r => r.filter(r => r.id !== id)); // optimistic
  try {
    await api.delete(`/records/${id}`);
    toast.success('Deleted');
  } catch (err) {
    setRecords(prev);             // revert
    toast.error('Delete failed');
    logger.error({ err, id }, 'Record deletion failed');
  }
}
```

```typescript
// ❌ BAD — user waits for email to send before getting a response

// ✅ GOOD — queue the email, respond immediately

```

## 14. FOUNDRIE AI ADDENDUM

### 14.3 Foundrie AI App Stack Rules

These rules apply to Foundrie AI itself. They do not force generated projects to use Next.js.

Context7 verified the App Router patterns used here.

- Use the root-level `app/` folder, not `src/app`.

Context7 verified the Clerk App Router baseline.

- Wrap the root app with `ClerkProvider`.

- Use server-side `auth()` for session identity in route handlers and server components.
- Verify Clerk webhooks with Svix before writing user records.
- Clerk owns authentication. Application code owns authorization.
- Every user-owned query must scope by authenticated local `user.id`.
- Ownership failures return 404, not 403.
- Do not build team workspaces, project-level RBAC, custom admin portals, RLS, ABAC, audit logs, or hardware-key admin controls unless a feature spec explicitly requires them.

### 14.5 Prisma and Neon Rules

Context7 verified the serverless PostgreSQL connection split.

- Neon Postgres is Foundrie's required relational database.
- Runtime Prisma Client uses pooled `DATABASE_URL` with the Neon pooler endpoint.
- Prisma CLI and migrations use direct `DIRECT_URL`.
- Every foreign key and hot filter path gets an index in the schema or raw SQL migration.
- List endpoints use cursor pagination, not offset pagination.
- List queries use `select` and avoid large JSON fields unless needed.
- Avoid N+1 query loops.
- Use transactions for multi-record consistency.
- Use raw SQL migrations for partial indexes and autovacuum tuning when Prisma cannot express the database feature.

### 14.6 Storage and Research Rules

- Vercel Blob stores files and generated artifacts.

- Use Blob client upload handlers only with owner checks, allowed content types, size limits, short-lived upload tokens, and database writes in the completion callback.
